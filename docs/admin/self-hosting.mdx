title: Self-hosting

This guide covers running gitcounsel on your own infrastructure: what you need, which settings are required, how to run it for local development, and how to bring up the full set of services with Docker.

## Prerequisites

- [Bun](https://bun.sh)
- Docker (for Postgres and the full stack)

## Configuration

Settings are read from environment variables. Copy `.env.example` to `.env` and fill in the required values.

| Variable                                                                                         | Purpose                                                                                                                                                            |
| ------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `DATABASE_URL`                                                                                   | Postgres connection string (the address and credentials for your database).                                                                                        |
| `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`                                                          | Authentication (sign-in).                                                                                                                                          |
| `ENCRYPTION_KEY`                                                                                 | Any non-empty string. A scrypt KDF turns it into a 32-byte AES-256-GCM key, used to encrypt user secrets (LLM keys, MCP credentials) so they are stored scrambled. |
| `COURTLISTENER_API_TOKEN`                                                                        | Enables the built-in legal-research tools.                                                                                                                         |
| `S3_ENDPOINT`, `S3_ACCESS_KEY`, `S3_SECRET_KEY`, `S3_REGION`, `S3_BUCKET`, `S3_FORCE_PATH_STYLE` | S3-compatible object storage (the file store for documents). The defaults target Cloudflare R2.                                                                    |

<Callout type="warn">
  Object storage is **required**. Set up an R2 bucket or any S3-compatible endpoint. There is no
  local-filesystem fallback (gitcounsel will not store files on the server's own disk instead).
</Callout>

### LLM provider keys

Provider keys (Claude / Gemini / OpenAI / OpenRouter) are usually added in-app under Settings, where they are encrypted and loaded at runtime. You can also set a server-wide fallback in env:

```bash
ANTHROPIC_API_KEY=...
OPENAI_API_KEY=...
GEMINI_API_KEY=...
OPENROUTER_API_KEY=...
```

A user's own key always **overrides** the server key. Pick the model per chat message or per review run. OpenRouter is the zero-data-retention default (the provider that, by default, keeps and trains on nothing).

## Local development

`bun run dev` starts the supporting services (Postgres and the Docling sidecar, a helper service that extracts text from documents) in Docker, applies the database schema, then runs the app:

```bash
bun install
bun run dev   # deps in Docker + web on http://localhost:4280
```

Useful flags:

- `--skip-deps` — the supporting services are already running.
- `--skip-migrate` — skip applying the database schema.
- `--dry-run` — show what would run without running it.

To do the same steps by hand (equivalent to `--skip-deps --skip-migrate`):

```bash
docker compose up -d postgres     # Postgres with pgvector
cd packages/db && bun run migrate  # apply schema
turbo run dev                      # web on http://localhost:4280
```

## Full stack with Docker

```bash
docker compose up --build
```

This runs three services:

- **web** — the gitcounsel app.
- **postgres** — Postgres with pgvector (the database, with the extension used for AI search).
- **docling** — the document-extraction service (it pulls text out of uploaded files), reachable on the internal network only so the outside world cannot call it.

A one-time `migrate` service applies the database schema and then exits. The `web` service waits for it to finish before it starts serving, and the running app never changes the schema itself.
