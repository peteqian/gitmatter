---
title: Data retention
description: How long each kind of data is kept, and how automatic cleanups work.
---

gitmatter removes old data on a rolling basis. In short: every kind of data has a time limit, and once data is past its limit it gets deleted. There is no separate scheduler or cron job. These cleanups ("purges") run every time the app starts up. A deployment that stays up keeps sweeping over time, and one that restarts sweeps on startup. Each retention window (how long data is kept) is set with an environment variable, measured in days. Setting a window to `0` turns that purge off, which means the data is kept forever.

## Retention windows

| Data                     | Variable               | Default             | Measured from                      | Notes                                                                                                                                                                                                                        |
| ------------------------ | ---------------------- | ------------------- | ---------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Documents (soft-deleted) | —                      | 30 days             | `deleted_at`                       | "Soft-deleted" means marked deleted and hidden, but still recoverable. After the 30-day window, the database row and the S3 file bytes are hard-deleted (permanently removed). This is built in and not configurable by env. |
| Chat history             | `CHAT_RETENTION_DAYS`  | `0` (keep forever)  | Last activity (`chats.updated_at`) | Pinned chats are never purged. Messages are removed with their chat.                                                                                                                                                         |
| Audit events             | `AUDIT_RETENTION_DAYS` | `365`               | `audit_events.created_at`          | The security and operations log: logins, token and key lifecycle, OAuth, uploads.                                                                                                                                            |
| OAuth auth codes         | —                      | Purged once expired | `expires_at`                       | Short-lived single-use codes, removed as soon as they expire.                                                                                                                                                                |
| OAuth access tokens      | `TOKEN_RETENTION_DAYS` | `30`                | `revoked_at`                       | Only **revoked** tokens are purged. A token whose access has merely expired is kept while its refresh token is still valid.                                                                                                  |
| Static MCP tokens        | `TOKEN_RETENTION_DAYS` | `30`                | `revoked_at`                       | Only **revoked** tokens are purged after the window.                                                                                                                                                                         |

## What is never purged automatically

- **Live tokens** (access tokens that have not been revoked, so they have no `revoked_at` date) are never purged on a timer. Revoke a token to start its retention clock.
- **Bring-your-own-key provider keys** (the AI provider keys a firm supplies itself) have no expiry. They are removed only when the user deletes them.
- **Artifacts** (matters, clients, tabular reviews, workflows) and the **git commit history** (the record of every change) stay until the artifact itself is deleted. They are not on a retention timer.

## Operational notes

- Purges are idempotent (running one twice does no harm) and best-effort. If one fails, the failure is ignored so it never blocks startup, and the next sweep retries.
- If deleting a file from object storage genuinely fails (as opposed to the file already being gone), it is recorded in the audit log as `storage.delete_failed`.
- To enforce a stricter policy, lower the relevant `*_RETENTION_DAYS` value in your deployment environment and restart.
