title: AI data handling

Law-firm documents are privileged (legally confidential). This page explains which content leaves gitmatter, where it goes, and how to set up the strictest privacy posture each AI provider allows, so providers keep as little of your data as possible.

<Callout type="warn">
  Provider privacy policies change over time. Verify a provider's current policy before relying on
  it.
</Callout>

## What gets sent to a provider

When you run a chat turn or a tabular review, gitmatter sends the relevant prompt and document content to **one** AI provider: the one that serves the model you picked. Nothing goes to providers you did not select. CourtListener legal-research calls go to CourtListener, not to an AI provider.

## Whose key is used

A "key" is the credential that bills and identifies the account talking to an AI provider. For each provider, gitmatter picks the key in this order: **the user's own key first**, then the server-wide key set in env (the deployment-level setting), and otherwise none. The model picker shows which source is active (`user` or `env`), so it is never ambiguous.

With bring-your-own-key (the firm supplies its own provider key), requests are billed and logged under the firm's own provider account.

## Per-provider retention posture

This table shows how much each provider keeps your data. "Trains on API data" means the provider uses your content to improve its models. "Zero data retention (ZDR)" means the provider does not keep or train on your data at all.

| Provider          | Trains on API data?                                    | Default posture                                               | True zero data retention (ZDR)                                                                                                          |
| ----------------- | ------------------------------------------------------ | ------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| **OpenRouter**    | No                                                     | Zero prompt and completion logging by default (metadata only) | Self-serve, and the only out-of-the-box ZDR. It is a proxy, so privileged data passes a third party before reaching the model provider. |
| **Anthropic**     | No                                                     | Roughly 7-day abuse-log retention; never trains on API data   | Needs a separate enterprise ZDR agreement (requires approval).                                                                          |
| **OpenAI**        | No                                                     | 30-day abuse retention                                        | Org-level ZDR requires sales approval.                                                                                                  |
| **Google Gemini** | No on the **paid** tier (the free tier **does** train) | Paid / Vertex: no training, limited abuse logging             | Per-project ZDR request (requires approval). Reject free-tier keys.                                                                     |

## What gitmatter sends today

gitmatter sets these privacy flags (settings that tell the provider not to keep your data) by default:

- **OpenAI** — `store: false` (the provider does not save the response on its servers), and reasoning is sent encrypted and never persisted (never kept).
- **OpenRouter** — `provider.dataCollection: "deny"` (route only to endpoints that do not retain or collect prompt data).
- **Anthropic and Gemini** — no training on API data by default, so no extra flag is needed for the default posture.

### Optional hardening

- OpenRouter also supports `zdr: true`, which restricts routing to ZDR-certified endpoints only (only providers that promise to keep nothing). It is stricter than `dataCollection: deny`, but it can shrink the set of available models, so enable it deliberately rather than by default.
- Use a **paid** Gemini key only. The free tier trains on submitted data.

## Enabling true ZDR

- **OpenRouter** — self-serve. The `dataCollection: deny` default already avoids retention. Add `zdr: true` for strict ZDR routing.
- **Anthropic, OpenAI, and Google** — request an enterprise or per-project ZDR agreement with the provider. Until that is in place, they keep short-lived abuse logs.

## User controls

- Pick the model (and therefore the provider) per chat message and per review run.
- Add or remove provider keys in Settings &rarr; AI &amp; Models.
- Export or delete your organization's data anytime in Settings &rarr; Data &amp; Privacy. See also the [data retention policy](/admin/data-retention).
