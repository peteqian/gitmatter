---
title: Workflows
description: Save a set of steps once, then run them again and again.
---

A **workflow** is a saved, reusable set of steps for a task you do often. Think of it
as a recipe. Instead of typing the same instructions every single time, you set them
up once. After that, you can run them whenever you need them, on any matter.

Why this helps: the work gets done the same careful way every time. Nothing gets
forgotten, and everyone on the team follows the same routine.

## The kinds of workflows

- **Assistant workflows** — a list of steps that the assistant carries out one after
  another, in order. It is like handing it a checklist and watching it work through
  each item for you. For example: "First summarise the contract. Then list every key
  date. Then flag anything unusual."
- **Tabular review workflows** — a ready-made set of columns to pull out of your
  documents and lay out in a table, like a spreadsheet (see
  [Tabular review](/user-guide/tabular-review)). For example, columns for party names,
  governing law, and renewal date.
- **Document generation** — fill in a template and produce a finished Word document.

## Built-in workflows and your own

gitcounsel comes with some ready-made workflows so you have a starting point. You can
also build your own and shape them in a simple editor, much like writing a document.

You can share a workflow with colleagues by email. That way a whole team works from
the same set of steps, instead of each person doing it their own way.

## Running a workflow

Open a workflow and run it against a matter or against a chosen set of documents. An
assistant workflow runs its steps in order, and each step builds on the one before it.

As with everything in gitcounsel, the results and any changes are written down: it
keeps a note of who ran the workflow and when. So if anyone ever asks how a result was
produced, you can show exactly what happened.
